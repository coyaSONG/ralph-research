<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| operator `rrx run` input -> service dispatcher | CLI flags and current persisted state determine whether the runtime resumes or starts fresh |
| persisted run checkpoint -> cycle runner step execution | resuming from the wrong step can corrupt the repo or persisted state |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-04 | T | `src/core/engine/cycle-runner.ts` | mitigate | Resume only from checkpoint-aligned pendingAction values produced by Plan 02-01; do not infer missing steps. |
| T-02-05 | R | `src/app/services/run-cycle-service.ts` | mitigate | Make latest-only selection explicit and log/return repair-required reasons instead of silently scanning older runs. |
| T-02-06 | D | `src/cli/commands/run.ts` | mitigate | Keep fail-fast behavior when the latest run is manual-review-blocked or failed; only repair_required gets warning-plus-fresh-start semantics. |
</threat_model>
